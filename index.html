为什么需要这个入口

  • 把分散的数据集、标注平台和工程系统收口到一个稳定入口。
  • 让外部访问者先理解生态结构，再进入具体子项目，而不是直接落到某一页代码或某一台 demo 服务上。
  • 用明确的 release boundary 区分“已公开”“可演示”和“仍在收尾”的内容，保证公开叙事可信。

这个页面提供什么

  • 会议二维码落地页
  • UAIV 各组成部分的统一公开导航
  • 数据集、平台与基础设施的当前状态摘要
  • 核心公开链接的集中入口

UAIV 不是几个分散网页，而是一条连贯的数据生产链。

这套基座真正的价值，在于它把 数据产品、人工标注工作台 与 数据操作系统 三层打通，而不是只发布一个数据集、只做一个标注工具，或者只停留在实验性质的工程脚本上。

数据产品

数据产品层

UAIV-Real 与 LAMD-DATASETS 负责对外发布数据资产，承载论文、公开下载、数据卡和后续 benchmark 的入口。

人工工作台

人工工作台层

UAIV-Labeler 负责人工标注、复核、模型辅助和任务导出，是数据真正被生产出来的前台工作台。

数据操作系统

数据操作系统层

UAIV-Foundry 负责 manifest、QA、release readiness、dataset card、benchmark protocol 和失败回流，是整条数据生产链的后端组织层。

当前基座已经具备的能力

从对外展示的角度，这套基座现在已经不仅仅是“有一些图像和一些代码”，而是形成了较完整的低空数据生产证据链，可以支撑论文叙事、项目展示和后续 benchmark 组织。

标准化数据

标准化数据资产

UAIV-Real CityGov 已经形成标准化图像记录、任务标注、split、dataset card 和 portable release package，不再依赖零散文件夹来表达数据资产。

人工参与

真实生产工作流

UAIV-Labeler 已经服务真实多人协作标注，具备目标检测、OCR、事件 QA、Metadata 管理、模型辅助和人工复核等生产工作流能力。

发布逻辑

发布与审计逻辑

UAIV-Foundry 把 manifest、QA、release readiness、dataset card、benchmark protocol 和 failure analysis 明确组织起来，为后续公开发布与评测奠定基础。

三类直接证据

这里不是装饰性视觉，而是直接来自真实工作台、真实结构设计和真实数据包统计。它们共同说明这套基座已经具备从数据组织到人工生产再到发布准备的连续能力。

真实标注工作台

UAIV-Labeler 已经承载真实多人标注，不只是静态 demo 页面。它已经具备面向低空多任务生产的工作流组织能力。

完整操作系统结构

Foundry 不只是脚本集合，而是显式组织了数据 planning、QA、release、benchmark protocol 和 iteration 的系统化基础设施。

标准化发布口径

CityGov 已经不是“文件夹里几千张图”，而是有 source-image pool、task-ready records、双 split 和正式 schema 的可发布数据包。

后续方向

这个页面的主叙事仍然是当前已经存在的真实资产。未来路线只保留最必要的结构性方向，用来说明 UAIV 将如何从数据产品与工作台，继续走向更完整的 benchmark 与 release infrastructure。

近期

完成 UAIV-Real CityGov 的公开沉积、论文提交收口和 release links 整理，让 DataHub 上的数据入口更完整。

中期

长期

逐步形成 UAIV-Benchmark 与 QA benchmark 层，让数据产品、工作台和数据操作系统自然过渡到统一评测体系。
